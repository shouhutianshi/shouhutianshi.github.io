性能优化之雅虎军规

发表于 2017-03-11   |   分类于 性能   |  

如何让页面最快应该是web开发最关心的问题。之前看了一遍《雅虎35条军规》，今日再来总结一下！(挑重点，有些感觉用不上的就不总结了)

Content

  1. Make Fewer HTTP Requests
    1. Minimize HTTP Requests减少/最小化http请求数
      到终端组件数的一个方法就是简化页面设计。保持富内容的页面且能减少http请求，有以下几个技术：
      • Combined fles。合并文件：1)合并js;2)合并CSS。
      • CSS Sprites。雪碧图可以合并多个背景图片，通过background-image background-position来显示不同部分。
      • Inline images。使用data:url scheme来内连图片。
    2. Reduce DNS Lookups(不明白怎么减少)
      减少DNS查询。
    3. Avoid Redirects
      避免跳转。
      最浪费的跳转之一发生在urlw尾部slash（/）缺失
    4. Make Ajax Cacheable
      让Ajax可缓存。提高ajax性能的最重要的方法是让相应被缓存。可以添加一个Expires 或者是 Cache-Control Header。还有其他方式是：
    5. Post-load Components
      延迟加载组件。“什么是页面初始化必须的？”剩下的内容和嘴贱可以延迟。
      JavaScript是理想的（延迟）候选者。可以切分到onload时间之前和之后。比如驼房的js库可以延迟，或者是隐藏的内容，折叠起来的图片等。
    6. Preload Components
      预加载组件。预加载看起来与延迟加载相反，但它的确有个不同的目标。通过预加载你可以利用浏览器的空闲时间请求你将来会用到的组件。这样当用户访问下一个页面时，你会有更多的组件已经在缓存中，这样会极大的加快页面加载。有几种预加载类型：
      * 无条件预加载：一旦onload触发，你立即获取另外的组件。
* 有条件预加载：基于用户动作，你推测用户下一步会去哪里加载并加载相应组件。
* 预期的预加载：在发布重新设计的网站前提前加载。在旧网页预加载新网页的部分组件，那么切换到新网页时就不会是没有任何缓存了、
    7. Reduce the Number of DOM Elements
      减少dom数。一个复杂的页面意味着更多的内容要下载，以及更慢的dom访问。如果你的页面dom元素很多，那么意味着你可能需要删除无用的内容和标签来优化。
    8. Split Components Across Domains
      把组件分散到不同的域名允许你最大化并行下载数。由于DNS查询的副作用，最佳的不同域名数是2-4。
    9. Minimize the Number of ifames
      最小化iframe的数量。iframe允许html文档被插入到父元素文档。iframe的优点有：
      * 帮助解决缓慢的第三方内容的加载，如广告和徽章
* 安全沙盒
* 并行下载脚本
      iframe的缺点：
      * 即使空的也消耗资源和时间
* 阻塞了页面的onload
* 非语义化
    10. Not 404
      不要404。http请求是昂贵的，所以发出http请求但获得没用的响应是完全不必要的，并且会降低用户体验。一些网站会由特别的404页面提高用户体验，但这仍然会浪费服务器资源。特别坏的是当链接指向外部js但却得到404结果。这样首先会降低并行下载数，其次浏览器可能会把404响应体当做js来解析，试图从里面找出可用的东西。
  2. Server
    1. Use a Content Delivery Network
      使用CDN。用户接近你的服务器会减少响应的时间。把你的内容发布到多个，地理上分散的服务器可以让页面加载更快。
      最好首先分发你的静态内容。这不仅可以减少响应时间，用CDN还很容易来做。
    2. Add an Expires or a Cache-Control Header
      加Expires或者Cache-Control头部。这条规则有两个方面：
      * 对静态组件： 通过设置Expires头部来实现“永不过期”的策略。
* 对动态组件： 用合适的Cache-Control头部来帮助浏览器进行有条件请求。
    3. Gzip Components
      使用时用Gzip等压缩组件。http请求或响应的传输时间可以被前端工程师显著减少。终端用户的带宽，ISP，接近对等交换点等等没法被开发团队控制。但是，压缩可以通过减少http响应的大小减少响应时间。
      尽可能去gzip更多文本类型的文件。html，脚本，样式，XML和JSON等等都应该被gzip，而图片，PDF等等不应该被gzip，因为它们本身已被压缩过，gzip它们只是浪费CPU，甚至增加文件大小。
    4. Avoid Empty Image src
      避免空src的图片。空src属性的图片的行为可能跟你预期的不一样，它有两种形式：
      * html标签：
* js: var img = new Image();img.src="";
      两种都会造成同一种后果：浏览器会向你的服务器发请求。
      * IE，向页面所在的目录发请求
* Safari和Chrome，请求实际的页面
* Firefox不再发请求
* Opera遇到空图片src不做任何事
  3. cookie
    http cookie的使用有多种原因，比如授权和个性化。cookie的信息通过http头部在浏览器和服务器端交换。尽可能减小cookie的大小来降低响应时间。
    * 消除不必要的cookie
* 尽可能减小cookie的大小来降低响应时间
* 注意设置cookie到合适的域名级别，则其他子域名不会被影响
* 正确设置Expires日期。早一点的Expires日期或者没有会尽早删除cookie，优化响应时间。
  4. CSS
    1. Put Stylesheets at the Top
      把样式放在顶部。
    2. Avoid CSS Expressions
      避免CSS表达式。
    3. Choose over @import
      在IE用@import和把CSS放到页面底部行为一致，所以最好别用。
    4. Avoid Filters
    5. 避免使用IE过滤器。
      最佳做法是用PNG8优雅降级。
  5. JavaScript
    1. Put Scripts at the Bottom
      把脚本放在底部。
    2. Make JavaScript and CSS External
      使用外部JS和CSS。就是使用外联的方式引入JS和CSS
    3. Minify JavaScript and CSS
      压缩JS和CSS。压缩就是删除代码中不必要的字符来减小文件大小，从而提高加载速度。当代码压缩时，注释删除，不需要的空格、换行、tab也被删除。除了外部脚本和样式，内连的脚本和样式同样应该被压缩。
    4. Remove Duplicate Scripts
      删除重复的脚本。 在页面中引入相同的脚本两次会上海性能。这可能会超出你的预料，当确实引入重复脚本，会发出不必要的http请求和浪费js执行时间。
      避免多次一引入脚本的方法是使用RequireJS，哈哈
    5. Minimize DOM Access
      最小化DOM访问。用JS访问DOM元素是缓慢的，所以为了响应更好的页面，你应该：
      * 缓存访问过的元素的引用
* 在DOM树外更新节点，然后添加到DOM树
* 避免用JS实现固定布局
  6. Images
    1. Optimize Images
      优化图片。运行压缩工具压缩图片。
    2. Optimize CSS Sprites
      优化雪碧图。
      * 把图片横向合而不是纵向，横向更小
* 把颜色相近的图片合并到一张雪碧图。
* 合并时，间距不要太大。
    3. Don’t Scale Images in HTML
      不要在html中缩放图片。不要因为你可以设置图片的狂傲就去用比你需要的大得多的图片。比如你需要100×100，不要用500×500。
    4. Make favicon.ico Small and Cacheable
      favicon.ico小且可缓存。favicon.ico是在你的服务器根路径的图片。邪恶的是即使你不关心它，浏览器仍然会请求它。所以最好不要响应404.
